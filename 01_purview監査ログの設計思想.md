# Purview 監査ログの設計思想

Copilot Studio のエージェントと会話をした後、Power Platform 管理画面や Microsoft Purview の監査ログを開くと、会話内容が **"Redacted"**(黒塗り)と表示されることがあります。これは不具合ではなく、明確な設計思想に基づくものです。

## なぜ会話の中身が見えないのか

Purview の監査ログは、そもそも **コンプライアンス・ガバナンス目的** の仕組みです。「誰が・いつ・どのエージェントを使ったか」「どのリソースにアクセスしたか」という **アクセスの事実** を記録することが目的であり、会話の中身そのものを監視することは設計上の主目的ではありません。

これは他の Microsoft 製品でも一貫している考え方です。たとえばファイルへのアクセス監査ログも「誰がいつこのファイルを開いたか」は記録しますが、「ファイルの中身」は監査ログ自体には含まれません。

Purview 監査ログが実際に記録しているのは、会話の全文ではなく **トランスクリプトのスレッド ID のみ** です。これが "Redacted" 表示の正体です。

### 理由1:プライバシー・データ最小化の原則(Data Minimization)

会話の全文には、個人情報、機密情報、時には健康情報や財務情報など、非常にセンシティブな内容が含まれる可能性があります。

もし「アクセス頻度の高い、多くの管理者が閲覧できる仕組み」である監査ログに会話全文をデフォルトで含めてしまうと、次のようなリスクが生じます。

- 本来アクセス権限を持つべきでない管理者にまで、会話の中身が見えてしまう
- GDPR や HIPAA など各種プライバシー規制における「収集する個人データは目的達成に必要な最小限にする」というデータ最小化の原則に抵触しやすくなる

そのため、「会話があったという事実」と「会話の中身」を意図的に別レイヤーに分離し、中身へのアクセスには追加の権限(Content Explorer Content Viewer ロールなど)を必要とする設計になっています。

### 理由2:権限の分離(Separation of Duties)

これは意図的なガバナンス設計です。

- 監査ログを見る権限(Audit Reader / Audit Manager)
- 会話の中身を見る権限(DSPM for AI の Content Viewer 相当)

をあえて分けることで、「誰が使ったかは把握したいが、会話の中身までは見る必要がない」という一般的な管理者と、「センシティブな内容の調査が必要な、限定されたコンプライアンス担当者」とで、アクセスできる情報の粒度を分けられるようにしています。

### 理由3:Application Insights はそもそも監査目的の製品ではない

Application Insights は Azure Monitor というアプリケーション運用監視(APM)製品の一部で、元々「アプリのパフォーマンスやエラーを開発者が追跡するため」に作られたものです。Copilot Studio と連携できるのは、開発者・運用担当者が動作状況をデバッグ・監視できるようにするための機能拡張であり、コンプライアンス上の正式な監査証跡として設計されたものではありません。

そのため、Application Insights 側で会話の中身を記録できても、それは「監査ログの代替」ではなく「運用監視のための別ツール」という位置づけになります。実際、この2つは組織内でも管理する部署が異なることが多く、監査・コンプライアンス側はセキュリティ/コンプライアンスチームが、Application Insights 側は開発・運用(CoE)チームが持つ、という役割分担になっているケースが一般的です。

## まとめ

| | 目的 | 想定利用者 | 中身の保持 |
|---|---|---|---|
| Purview 監査ログ | 「誰が・いつ使ったか」の証跡 | 全社共通の管理者 | 原則含まない |
| DSPM for AI | コンプライアンス調査 | 限定されたコンプライアンス担当 | スレッドID経由で中身にアクセス可 |
| Application Insights | 運用監視・デバッグ | 開発・運用担当 | 設定次第で中身を記録可 |

「誰でも見られる仕組み」と「中身まで見られる限定的な仕組み」を分けることで、利便性とプライバシー保護のバランスを取っている、という理解で概ね正しいです。

## 補足:ユーザー入力とEntra ID認証の関係

Copilot Studio のログで「ユーザーが質問する前」ではなく「入力した直後」がイベントの起点になっているのは、「ユーザーがメッセージを送信した」というアクション自体が、Microsoft 365(Entra ID)の認証が完了していることの証拠になるためです。認証が完了していなければ、そもそもメッセージの送信自体が成立しません。
